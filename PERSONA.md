## ペルソナ1: 田中 翔（28歳）— スタートアップのAIエンジニア

**状況**

- 社員15名のスタートアップでLLMアプリ開発を一人で担当
- 社内向けRAGシステムを3ヶ月前に本番リリースしたが、「精度が良いのか悪いのかわからない」と言われている
- LangSmithを試したが月$39が稟議に通らなかった

**痛み**

- RAGの回答が改善しているのか退化しているのかわからない
- チャンク設計を変えるたびに「前より良くなったか」を手動で確認している
- Ragasは試したがコードを書くだけでUIがないので、非エンジニアのPdMに結果を共有できない

**RAG評価ダッシュボードの使い方**

- `pip install rag-eval` して既存のRAGに3行追加
- PdMにダッシュボードのURLを共有して「ここで品質確認してください」と渡す
- 無料枠で十分

**リーチ方法**: Zenn記事・GitHub・X（技術系）

**優先度**: ★★★（最初のターゲット）

## ペルソナ2: 鈴木 美咲（35歳）— 中堅SIerのAI推進担当

**状況**

- 社員500名のSIerで「AI推進室」を立ち上げたばかり
- 顧客向けにRAG構築の提案をしており、PoC成功率の低さに悩んでいる
- 顧客のデータを外部サービスに送ることができないプロジェクトが多い

**痛み**

- PoC後に「本当に精度が出ているのか」を客観的な数字で説明できない
- LangSmithはセルフホスト不可なのでセキュリティ審査が通らない
- Langfuseはセルフホストできるが、サーバー構築に工数がかかり展開が遅い

**RAG評価ダッシュボードの使い方**

- Docker Composeでオンプレ環境に展開
- 顧客に「評価スコアが0.82→0.91に改善しました」とレポート提出
- セルフホスト型を選択（データが外に出ない）

**リーチ方法**: Qiita・技術勉強会・LinkedIn

**優先度**: ★★（セルフホスト需要・単価が上がる）

## ペルソナ3: Alex Chen（32歳）— 海外のフリーランスML Engineer

**状況**

- フリーランスでRAGシステムの構築・改善を受託
- 複数クライアントのRAGを同時に管理している
- 既にLangSmithを使っているが、クライアントに見せるためだけにシート追加が必要で割高

**痛み**

- クライアントごとにダッシュボードを分けたいが、LangSmithはシート課金でコストが積み上がる
- クライアントのデータが同一テナントに混在するのが気になる
- 日本語RAGの評価は英語ベースのメトリクスだと精度が出ない

**RAG評価ダッシュボードの使い方**

- SaaS型でAPIキーをクライアントごとに発行
- クライアントにダッシュボードアカウントを渡して「自分で確認できる」状態にする
- ユーザー数無制限課金なのでクライアントが増えてもコストが読める

**リーチ方法**: HackerNews・英語Zenn記事・GitHub README

**優先度**: ★（将来の国際展開層）
